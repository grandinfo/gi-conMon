## 1. 架构目标与约束

### 1.1 质量属性目标

| 属性 | 目标 | 量化指标 |
|------|------|---------|
| **可用性** | 系统自身高可用 | 控制端 99.99%，探针自治不间断 |
| **性能** | 大规模并发探测 | 单节点 100,000 QPS，P99 < 10ms |
| **可扩展性** | 水平弹性扩展 | 探针节点与目标数量线性扩展 |
| **可维护性** | 运维友好 | 配置热重载，零停机升级 |
| **安全性** | 企业级安全 | TLS 全链路，RBAC，国密可选 |
| **可观测性** | 自身可观测 | Prometheus 自监控，结构化日志 |

### 1.2 设计约束

- 核心探测路径必须与控制平面解耦，探针断网后仍可独立工作
- 单一二进制交付，嵌入式场景无外部依赖
- 插件接口稳定，版本向后兼容
- 敏感配置（密码、Token）不得以明文落盘

## 3. 核心组件

### 3.1 组件清单

| 组件 | 职责 | 进程模式 |
|------|------|---------|
| **conmon-server** | API 网关、状态汇聚、告警决策、配置管理 | 集群多活 |
| **conmon-probe** | 执行探测任务，上报结果 | 边缘部署，可独立运行 |
| **conmon-alerter** | 告警规则评估、通知发送、升级管理 | 可内嵌 server 或独立部署 |
| **conmon-reporter** | SLA 计算、报表生成 | 批处理，按需启动 |
| **conmon-ui** | Web 控制台前端（静态资源） | 由 server 托管或独立 CDN |
| **conmon** (CLI) | 命令行管理工具 | 客户端，无常驻进程 |

## 4. 数据流设计

### 4.1 探测数据流

```
探针节点                    控制端                     存储层
   │                          │                          │
   │  1. 调度器触发探测任务    │                          │
   ├─────────────────────►    │                          │
   │  2. 执行协议探测          │                          │
   │  3. 获取探测结果          │                          │
   │  (延迟、状态码、错误信息)  │                          │
   │                          │                          │
   │  4. gRPC 流式上报结果     │                          │
   ├─────────────────────────►│                          │
   │                          │  5. 状态机处理            │
   │                          │  (对比历史状态，判断变更)  │
   │                          │                          │
   │                          │  6. 写入时序数据          │
   │                          ├─────────────────────────►│
   │                          │  (延迟、丢包率、状态值)    │
   │                          │                          │
   │                          │  7. 状态变更？            │
   │                          │  ├─YES─► 发布事件到 Bus   │
   │                          │  └─NO──► 仅更新指标       │
```

### 4.2 告警触发流

```
事件总线                  告警引擎                    通知渠道
   │                         │                          │
   │  status_changed 事件    │                          │
   ├────────────────────────►│                          │
   │                         │  1. 规则评估              │
   │                         │  2. 依赖拓扑检查          │
   │                         │  (上游 DOWN？→ 抑制)      │
   │                         │  3. 去重检查              │
   │                         │  4. 模板渲染              │
   │                         │                          │
   │                         │  5. 并发发送多渠道        │
   │                         ├─────────────────────────►│
   │                         │                          │
   │                         │  6. 记录发送结果          │
   │                         │  7. 启动升级计时器        │
   │                         │     (5min → L2 → L3)     │
```

## 6. 技术选型

### 6.1 核心技术栈

| 模块 | 技术选型 | 选型理由 |
|------|---------|---------|
| **服务端语言** | Go 1.22+ | 高并发、低内存、单二进制交付、系统级网络编程 |
| **内部 RPC** | gRPC + Protocol Buffers | 高性能双向流，探针↔控制端通信 |
| **API 框架** | Gin / Fiber | 高性能 HTTP，低延迟 API 服务 |
| **配置中心** | etcd v3 / Consul | 分布式 KV，Watch 机制，Leader 选举 |

## 7. 高可用设计

### 7.1 控制端高可用

```
                   VIP (虚拟 IP)
                       │
             ┌─────────┴─────────┐
             │                   │
        ┌────▼────┐         ┌────▼────┐
        │ Server1 │◄──────►│ Server2 │   Raft 选主（通过 etcd）
        │  (主)   │         │  (从)   │   主节点处理写入
        └─────────┘         └─────────┘   从节点处理只读请求
             │
        ┌────▼────┐
        │ Server3 │
        │  (从)   │
        └─────────┘
```

- 使用 etcd 进行 Leader 选举，主节点负责告警决策，从节点处理 API 只读查询
- 主节点故障后，etcd 选举新主节点，切换延迟 < 5s
- 所有节点共享同一 PostgreSQL（主从复制）

### 7.2 探针自治机制

```
正常状态:
  探针 ──gRPC──► 控制端  (实时上报)

控制端不可达:
  探针 ──✗──► 控制端
    │
    ├── 本地配置缓存继续执行探测
    ├── 结果写入本地环形缓冲区 (最多 10 万条)
    └── 每 30s 重试连接控制端

恢复连接后:
  探针 ──gRPC 批量补传──► 控制端
    └── 按时间戳顺序补传历史结果
    └── 控制端回放状态机，补算历史事件
```

### 7.3 存储高可用

| 层次 | 策略 |
|------|------|
| PostgreSQL | 主从流复制 + 自动 Failover（Patroni 或 CloudSQL） |
| InfluxDB | 集群模式 / 单机版 + 定期快照 |
| etcd | 3 节点 Raft 集群，单节点故障不影响可用性 |
| 对象存储 | 跨可用区冗余，99.999999999% 持久性 |

## 9. 性能架构

### 9.1 探测并发模型

```
调度器（单 goroutine）
    │
    │  探测任务入队
    ▼
优先队列（3 个级别）
    │
    ├── high  优先队列 (P0 目标，容量 1000)
    ├── normal 普通队列 (默认，容量 10000)
    └── low   低优先级队列 (批量检查，容量 50000)
    │
    ▼
goroutine 池（信号量控制并发数 = concurrency 参数）
    │
    ├── goroutine-1 ──► ICMP 探测
    ├── goroutine-2 ──► HTTP 探测
    ├── goroutine-N ──► DNS 探测
    └── ...（最多 concurrency 个并发，默认 100）
```

### 9.2 写入性能优化

| 优化手段 | 说明 |
|---------|------|
| **批量写入** | 时序数据攒批（batch_size=1000 或 flush_interval=500ms）写入 InfluxDB |
| **异步写入** | 探测结果写入非阻塞，先进内存队列，后台协程异步落盘 |
| **写入分离** | 时序数据写 InfluxDB，事件数据写 PostgreSQL，各自优化 |
| **连接池** | PostgreSQL 连接池 max=50，InfluxDB 连接池 max=20 |
| **本地聚合** | 探针本地聚合 1 分钟内的延迟统计（P50/P95/P99），减少上报量 |

### 9.3 读取性能优化

| 场景 | 优化手段 |
|------|---------|
| 全局状态快照 | 内存缓存（TTL=1s），避免实时查库 |
| 目标列表 API | 分页 + 索引（tags GIN 索引，host B-tree 索引） |
| 延迟时序查询 | InfluxDB Flux 查询 + 降采样（1min/1h/1d 多级聚合） |
| SLA 计算 | 预计算缓存，每小时增量更新 |
| 实时 Dashboard | WebSocket 推送，状态变更时主动推送，无需客户端轮询 |

## 附录：架构决策记录（ADR）

### ADR-001：控制平面与数据平面分离

**背景**：探针部署在各地域，网络不稳定，需要保证探测不中断。

**决策**：探针节点实现本地配置缓存和结果缓冲，与控制端解耦。

**结果**：探测平面可在控制端故障时独立工作最长 24 小时。

### ADR-002：选用 Go 而非 Python/Java

**背景**：需要单二进制交付，支持嵌入式 ARM 设备，高并发低内存。

**决策**：Go 1.22，交叉编译支持 linux/amd64、linux/arm64、darwin/amd64、windows/amd64。

**结果**：二进制大小约 20MB（静态链接），内存占用约 50MB（5000 目标场景）。

### ADR-003：配置中心选用 etcd 而非数据库

**背景**：配置变更需要秒级同步到所有探针节点，数据库轮询延迟太高。

**决策**：etcd Watch 机制，配置写入后 Watch 回调链路延迟 < 200ms。

**结果**：配置全局生效时间 < 1s。

### ADR-004：规则引擎选用 CEL 而非 Lua/JavaScript

**背景**：告警规则需要用户自定义，但必须安全、无副作用、高性能。

**决策**：Google CEL（Common Expression Language），沙箱执行，无文件/网络访问，规则评估 P99 < 1ms。

**结果**：规则表达能力满足需求，无安全风险。
